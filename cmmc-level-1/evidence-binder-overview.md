# CMMC Level 1 Evidence Binder Overview

## Purpose

A CMMC Level 1 evidence binder organizes the policies, procedures, screenshots, logs, checklists, and management records that demonstrate basic safeguarding practices for Federal Contract Information (FCI). The binder gives leadership, reviewers, and contract stakeholders a clear way to see what controls are in place, what evidence supports those controls, and what items still require improvement.

For A+ Enterprise LLC, the evidence binder supports audit-ready public operations, compliance documentation, CMMC readiness, NIST SP 800-171 control mapping, incident response documentation, vulnerability management reporting, and automation-enabled execution through BranchOS.

## Public CMMC Statement

A+ Enterprise LLC maintains a CMMC Level 1 self-assessment in SPRS under UID S100053243 for FCI/basic safeguarding alignment under FAR 52.204-21. Level 2 readiness will be evaluated only where CUI or contract-specific requirements apply.

## Binder Sections

### 1. Governance and Scope

This section defines what systems, users, services, data types, and business processes are included in the Level 1 readiness effort. It identifies whether the environment handles FCI, CUI, public information, or internal business records.

Example evidence:

- System boundary statement
- Data handling summary
- Roles and responsibilities matrix
- Public-safe company compliance statement
- Contract requirement review checklist

### 2. Access Control

This section shows how the organization limits system access to authorized users and appropriate business functions.

Example evidence:

- User access review checklist
- Account approval records
- Multi-factor authentication settings
- Privilege review notes
- Offboarding checklist

### 3. Awareness and Training

This section records how personnel are trained to recognize cybersecurity responsibilities, phishing risks, data handling boundaries, and incident reporting expectations.

Example evidence:

- Security awareness training record
- Phishing awareness checklist
- Acceptable use acknowledgment
- Incident reporting guide
- Annual training schedule

### 4. Incident Response

This section documents how incidents are reported, triaged, contained, escalated, and reviewed after resolution.

Example evidence:

- Incident response playbook
- Severity matrix
- Incident intake form
- Evidence preservation checklist
- Lessons-learned template

### 5. Vulnerability Management

This section demonstrates how systems are reviewed for security weaknesses and how remediation work is tracked to completion.

Example evidence:

- Sample vulnerability scan report
- Remediation tracking log
- Patch verification notes
- Risk acceptance record
- Follow-up scan summary

### 6. Configuration and Change Management

This section explains how security-relevant configuration changes are reviewed, documented, and validated.

Example evidence:

- Configuration baseline checklist
- Change request log
- Approved software list
- Secure settings checklist
- Change validation notes

### 7. Audit Readiness and Continuous Improvement

This section connects the evidence binder to ongoing compliance operations. It should show that compliance is maintained over time rather than assembled only when a review is pending.

Example evidence:

- Monthly compliance review notes
- Control status tracker
- Open action register
- Management review summary
- BranchOS dashboard export or summary

## How the Binder Supports Audit Readiness

An effective evidence binder reduces review friction by keeping control narratives, supporting artifacts, ownership, and status in one organized location. It helps leadership answer three practical questions:

1. What safeguarding requirement applies?
2. What process or control is used to meet it?
3. What evidence shows that the process is operating?

The binder should not contain private credentials, bank information, client-sensitive files, protected health information, Social Security numbers, VA file numbers, or confidential contract records. Sensitive source records should remain in controlled private storage with only sanitized summaries used for public portfolio examples.
